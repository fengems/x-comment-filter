## 二、核心技术原理（这类扩展的通用骨架）

所有同类产品，无论功能多花哨，**底层都是同一套技术栈**：

### 2.1 三层架构

```
┌─────────────────────────────────────────┐
│  Content Script（注入 x.com 页面）        │  ← 核心战场
│  ┌───────────────────────────────────┐  │
│  │ MutationObserver 监听 DOM 变化     │  │
│  │ → 定位推文/评论节点（selector）     │  │
│  │ → 提取文本/用户名/属性             │  │
│  │ → 判定（关键词/规则/AI）           │  │
│  │ → 动作（隐藏/折叠/点击 block）     │  │
│  └───────────────────────────────────┘  │
├─────────────────────────────────────────┤
│  Background (Service Worker, MV3)        │  ← 协调、定时任务、跨 tab
│  存储、alarm 定时同步、右键菜单、限速队列  │
├─────────────────────────────────────────┤
│  Popup / Options 页                      │  ← 设置 UI
│  规则增删改、统计、导入导出              │
└─────────────────────────────────────────┘
```

### 2.2 DOM 监听：MutationObserver（必备）

X 是重 SPA，内容靠 React 动态渲染 + 虚拟列表无限滚动。**几乎所有扩展都用 MutationObserver** 监听 `document.body`：

```js
// 标准写法
const observer = new MutationObserver((mutations) => {
  // 遍历 addedNodes，找新增的推文节点
});
observer.observe(document.body, { childList: true, subtree: true });
```

**关键经验（踩坑后才懂的细节）：**

1. **配 `childList + subtree`，不要配 `characterData`**（避免文本编辑型 DOM 抖动，参考 x-comment-blocker）
2. **必须处理"虚拟列表复用"**：X 会复用容器只换内部内容，只看 `addedNodes` 会漏判。x-comment-blocker 对 `mutation.target` 也做了 `closest('[data-testid="cellInnerDiv"]')` 回捞——这是踩过坑才会加的
3. **批处理 + 防抖**：MXGA 用 600ms 防抖；x-comment-blocker 用 `pendingTweets` Set 在每轮回调末尾批量处理一次；都**没有逐条同步处理**（性能杀手）
4. **加定时兜底**：MXGA 额外 `setInterval(scan, 4000)`，防止不滚动时的边缘情况
5. **SPA 路由切换要清缓存**：监听 URL 变化，切页清掉去重缓存（否则跨页计数污染）
6. **isExtensionAlive 检查**：扩展更新/卸载后 observer 还在跑会抛错，入口检查 `chrome.runtime?.id`

### 2.3 节点定位：依赖 `data-testid`（抗改版核心）

X 的 className 是 CSS-in-JS 生成的随机 hash（如 `css-1dbjc4n`），**完全不可靠**。但 `data-testid` 是开发测试锚点，相对稳定。几乎所有扩展都用：

| selector | 含义 |
|---|---|
| `[data-testid="tweet"]` 或 `article` | 一条推文/回复的根 |
| `[data-testid="cellInnerDiv"]` | 时间线/评论流的**滚动列表项容器**（隐藏整条用这个） |
| `[data-testid="tweetText"]` | 推文正文（提取文本，**当前最稳的文本锚点**） |
| `[data-testid="User-Name"]` | 推文头部用户名区（挂徽标用） |
| `[data-testid="placementTracking"]` | 推广推文外层 |
| `[data-testid="socialContext"]` | 转推/社交上下文 |

**抗改版的多重策略（Control Panel for Twitter 的精华）：**

1. **能读 React state 就不靠 DOM**（最强护城河，但要跑在 main world）：通过 `__reactFiber$` / `__reactProps$` 遍历拿 Redux store，得到权威推文数据（`full_text`、`isBlueVerified` 等），比扒 DOM 文本稳得多
2. **优先用 `data-testid`，几乎不依赖 className**
3. **SVG path 当指纹**：图标 SVG 的 `d` 属性前缀做识别（转推图标、推广图标等），比 className 稳定
4. **多 selector 兜底链**：twitter-blocker 的 `extractReplyText` 用 4 个 selector 依次尝试
5. **防御性编程 + 详尽日志**：每个假设失败都有日志，便于用户上报

> ⚠️ 即便如此，selector 失效仍是这类扩展的**永恒痛点**。作者们都在持续追改。Control Panel for Twitter 的代码里甚至有个苦笑式计数：`// Add 1 every time this gets broken: 6`。

### 2.4 隐藏内容的手段（三种，体验递进）

| 手段 | 实现 | 体验 | 谁在用 |
|---|---|---|---|
| **直接 display:none** | `node.style.display = 'none'` 或加 class + CSS | 最简单，但用户不知道屏蔽了什么、无法恢复 | x-comment-blocker、xModerator（hide 模式） |
| **模糊 + "Show anyway"** | `filter: blur(5px)` + 蒙层按钮 | 渐进式，可还原 | xModerator（blur 模式） |
| **折叠占位卡**（推荐） | 替换为"已屏蔽 N 条 [展开]" | 体验最好，可批量展开 | 暂无成熟开源实现（机会点） |

> 隐藏时**不要 `removeChild`**，要 toggle class + CSS——可逆、不破坏 React 的 reconciliation、便于调试（Control Panel for Twitter 的关键设计）。

### 2.5 判定方式：三条技术路线

#### 路线 A：关键词/规则匹配（最主流，本地、零成本）
- **做法**：提取文本 → 正则/子串匹配 → 命中隐藏
- **优点**：零成本、零隐私、即时、可自托管词库
- **缺点**：无语义理解，碰上改写/谐音/拼音/emoji 绕过就无力
- **代表**：x-comment-blocker、xModerator、绝大多数商业扩展
- **关键反绕过技巧**（x-comment-blocker 的精华）：
  - 清洗零宽字符：`/[\u00AD\u200B-\u200F\u202A-\u202E\u2060-\u206F\uFEFF]/g`（垃圾信息常用 `免\u200b费` 绕过）
  - 用户名标点清洗：`[\s_.-]+` 去掉（防 `免.费`）
  - 把 `<img alt>`（emoji）也算进文本
- **坑**（xModerator 的教训）：纯 `text.includes(keyword)` 子串匹配会大量误伤——`sex` 命中 "Sussex"、`cum` 命中 "document"、`rape` 命中 "scrape"、`ad` 几乎匹配一切。**必须用词边界 `\b` 或 token 化**

#### 路线 B：云端 LLM 分类（最智能，有成本）
- **做法**：回复文本发到后端（Cloudflare Worker）→ 调 OpenAI → 返回 `{isMatch, confidence, reason}`
- **优点**：语义理解强、可判改写、零配置（规则写在 prompt 里）
- **缺点**：有 API 成本、有隐私争议（文本上传第三方）、串行延迟、中心化单点、误判不可逆（自动 block 时）
- **代表**：twitter-blocker（OpenAI）、MXGA（服务端 LLM + 人工确认）
- **关键架构**：API key 放 Worker 环境变量（不能内置扩展里会泄露）；扩展只做 DOM 提取和动作
- **必加**：去重缓存（用户名+文本片段 key）、每日配额、误判 undo、白名单、dry-run 模式

#### 路线 C：本地 ML 推理（折中，浏览器内）
- **做法**：用 [Transformers.js](https://huggingface.co/docs/transformers.js) / ONNX Runtime Web 在浏览器内跑 DistilBERT 级分类模型
- **优点**：隐私好（不上传）、语义理解
- **缺点**：模型要训练/转换、首次加载慢、占内存、可能卡主线程
- **参考**：[From PyTorch to Browser（toxicity 分类全流程）](https://bandarra.me/posts/from-pytorch-to-browser-a-full-client-side-solution-with-onnx-and-transformers-js)
- **实战未见成熟推特扩展采用**（可能因上述缺点）

### 2.6 屏蔽动作（要不要"真"屏蔽）

- **纯隐藏**（推荐起步）：只在本端 `display:none`，X 无感，零风险
- **折叠**：占位卡，可还原
- **调 X API 真 mute/block**（MXGA/twitter-blocker）：用用户自己的登录态调 `/i/api/1.1/mutes|blocks/create.json`。**必须加全局限速队列**（MXGA：跨 tab Web Locks 串行、~1.2s 间隔 + 抖动、每 45 次冷却、429 退避），否则触发 X 风控（Ghost Ban/冻结）
- **模拟点击**（twitter-blocker）：X 的 React 事件系统对 `el.click()` 不生效，要手动派发完整事件序列 `pointerdown→mousedown→pointerup→mouseup→click`，且 `composed: true` 穿透 Shadow DOM

### 2.7 数据/规则存储

- **`chrome.storage.local`**：大数据、统计、设备本地（x-comment-blocker、MXGA）
- **`chrome.storage.sync`**：小配置跨设备同步，配额 100KB（xModerator 的设置走 sync）
- **云端词库热更新**（x-comment-blocker 的聪明方案）：词库存 GitHub 仓库 `keywords.txt`，扩展用 **GitHub Contents API + ETag 做增量同步**（`If-None-Match`，304 跳过），6 小时一次。零后端成本
- **社区共建公榜**（MXGA 的进阶方案）：服务端 D1 真源 → 编译成 Bloom filter / 名单 JSON 随包分发；扩展端零网络。带治理（AI 永不单独公开、人工确认、git history 当审计日志）

## 三、产品功能与交互设计参考

### 3.1 过滤维度全景（从各产品归纳）

**按内容**：关键词/正则、特殊字符（花体字垃圾）、emoji、外链/可疑域名
**按用户**：用户名黑名单、handle、头像、注册时间（新号）、粉丝数、是否有蓝V
**按结构**（xModerator 的方向，但未接线）：全大写比例、标点比例、重复字符、多 URL
**按位置/类型**：评论区 vs Feed、主推文 vs 回复、转推/引用、推广推文、通知、UI 元素（侧边栏/推荐关注）
**按行为**：导流模板结构（短中文 + @mention + emoji + 性暗示）

### 3.2 值得借鉴的交互设计

1. **总开关**（toolbar toggle）——Control Panel for Twitter、x-comment-blocker 都有
2. **屏蔽统计 + 历史**——"已屏蔽 N 条" + 查看命中记录（含 user/text/reason/time）
3. **逐词粒度开关**（xModerator 的亮点）——内置词表每个词可独立开关，不只能整类开关
4. **模糊 + "Show anyway"**（xModerator）——渐进式屏蔽，比硬删除友好
5. **右键菜单"添加到屏蔽词"**（x-comment-blocker）——选中文字一键加词，很贴心
6. **自动保存 + 实时响应**（x-comment-blocker）——改设置立即生效，靠 `chrome.storage.onChanged`
7. **5 秒可撤销窗口**（MXGA）——命中后先挂"5秒后隐藏 [撤销]"，到点才真隐藏
8. **导入导出 JSON**——规则集可迁移
9. **行内徽标**（MXGA）——在用户名旁挂"首发/命中名单/待定"徽标，hover 显示 reasons
10. **三类徽标视觉区分**：fresh（首发，hollow ring）/ known（命中名单，实心点）/ pending（倒计时，pulse）

### 3.3 用户最关心的痛点（调研归纳）

| 痛点 | 现状 |
|---|---|
| 被屏蔽内容看不到、不可恢复 | 多数产品直接 display:none，无折叠 |
| 关键词误伤（子串匹配） | xModerator 严重误伤，是普遍问题 |
| 规则不能按维度精细配置 | 多数产品规则模型太扁平（所有词共享全局开关） |
| X 改版后插件失效 | 永恒痛点，靠持续维护 |
| 隐私（数据上传） | 云端方案有争议，本地方案最佳 |
| 词库需要持续更新 | GitHub CDN / 社区公榜两种解法 |

## 四、关键结论（指导自研）

### 4.1 应该借鉴的精华

| 来源 | 借鉴点 |
|---|---|
| **x-comment-blocker** | Observer + pendingTweets 批处理 + cellInnerDiv 虚拟列表回捞；评论区判定三件套（URL 状态页 + 主推文判定）；零宽字符清洗；`__cbxQuickHash + filterVersion` 缓存；GitHub Contents + ETag 词库热更新 |
| **Control Panel for Twitter** | page-world 注入读 React state（抗改版最强）；加 class 不删节点；批处理 DOM 写入；observer 命名集中管理防泄漏；多 selector 兜底 + SVG path 指纹 |
| **MXGA** | fiber 直读用户 profile（bio/注册时间零请求获取）；虚拟化列表对齐（WeakMap 以账号为 key）；零默认网络姿态；限速队列调 X API；5 秒可撤销；Shadow DOM 渲染自有 UI；Bloom filter 公榜 |
| **twitter-blocker** | simulateClick 5 事件序列 + `composed:true`；去重 key 设计；结构化 JSON LLM 输出；每日配额 |
| **xModerator**（反面教材） | 双存储设计思路（sync 设置 + local 统计）；逐词开关；模糊+还原 UI —— 但**要避开它的子串匹配误伤、storage key 不一致 bug、权限缺失** |

### 4.2 应该避免的坑

1. ❌ 纯 `text.includes(keyword)` 子串匹配 → 必然误伤，用词边界/token 化
2. ❌ observer 回调里逐条同步处理 → 性能崩，用批处理 + 防抖
3. ❌ 只监听 `addedNodes` 不管虚拟列表复用 → 漏判
4. ❌ 直接 `removeChild` → 不可逆，破坏 React，用 toggle class
5. ❌ selector 单点无兜底 → X 改版即失效
6. ❌ 把 API key 内置扩展 → 泄露，放服务端
7. ❌ 自动 block 无 undo/白名单 → 误判误封真实用户
8. ❌ storage key 前后端不一致（xModerator 的真实 bug）→ 数据不通
9. ❌ 用 className 定位 → 随机 hash 不可靠，用 data-testid
10. ❌ 上传用户数据到第三方 → 隐私争议，本地优先

### 4.3 市场空白（咱们的差异化机会）

经调研，**目前没有一个开源/商业扩展同时做到**：

1. ✅ **折叠占位卡 UI**（"已屏蔽 N 条 [展开]"）——多数直接 display:none
2. ✅ **多维规则引擎**（每条规则可独立设 scope/mode/白名单）——多数规则模型太扁平
3. ✅ **本地 + 云端混合判定**（关键词预筛 + 可选 LLM 精判）——要么纯关键词要么纯云端
4. ✅ **完善的误伤防护**（白名单、词边界、可撤销、dry-run）——多数缺失
5. ✅ **可自托管词库 + 社区共建**——x-comment-blocker 有云端词库但无共建，MXGA 有共建但复杂

**这就是咱们插件的核心定位空间。**
