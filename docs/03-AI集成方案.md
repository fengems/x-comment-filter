# AI 集成方案：三模式架构 + 智能规则挖掘

> 基于 `01-调研报告.md`、`02-技术方案.md`。
> 核心：**本地硬编码规则实时判断 + 大模型（本地/云端）做智能规则挖掘**，而不是让大模型实时逐条判断。

---

## 一、核心设计哲学（先讲清楚为什么这么设计）

### 1.1 关键洞察：大模型不该实时跑，但该"教"规则

经调研，大模型（无论 Chrome 内置 / WebLLM / 云端 API）**都不适合实时逐条过滤评论**：

| 方式 | 实时逐条的问题 |
|---|---|
| Chrome 内置 Gemini Nano | 推理慢（数百 ms/条）、模型不可选、用户未必开启，逐条调卡顿 |
| WebLLM 本地小模型 | 首次下载几百 MB、占内存 3-10GB、逐条推理延迟高 |
| 云端 API | 有成本、有延迟、有隐私（上传文本）、有配额、串行慢 |

**而规则匹配（关键词/正则）实时逐条零成本、零延迟**，缺点是"只会过滤已知模式"。

### 1.2 正确分工：模型当"老师"，规则当"学生"

```
┌─────────────────────────────────────────────────────────┐
│  实时层（每条评论都过，零成本）                            │
│  规则引擎（关键词/正则/用户名/结构特征）→ 折叠/放行        │
│  ↑ 规则来自哪？                                            │
├─────────────────────────────────────────────────────────┤
│  智能层（后台采样，不实时）                                │
│  大模型分析"规则没命中但可疑"的评论 → 提炼新规则/模式      │
│  → 经用户确认 → 写入规则库 → 实时层立刻变强                │
└─────────────────────────────────────────────────────────┘
```

这就是你说的"**智能检测 → 发现新类型 → 整理规则**"——一个**自进化的过滤系统**，而不是一个"每条都问大模型"的笨系统。

### 1.3 三种"判定后端"（用户可选，互为补充）

| 模式 | 实时性 | 隐私 | 成本 | 智能度 | 用途 |
|---|---|---|---|---|---|
| **Mode 1：纯规则**（本地硬编码 + 用户规则 + 云端词库） | ⚡ 实时 | 🟢 零网络 | 🆓 免费 | ⭐⭐ 基础 | 默认模式，所有用户 |
| **Mode 2：本地大模型辅助**（Chrome 内置 AI / WebLLM，opt-in） | 🐢 后台 | 🟢 零网络 | 🆓 免费 | ⭐⭐⭐⭐ 挖掘新模式 | 高级用户、离线 |
| **Mode 3：云端大模型辅助**（DeepSeek/ChatGPT 等，opt-in） | 🐢 后台 | 🟡 上传文本 | 💰 按 token | ⭐⭐⭐⭐⭐ 最强 | 愿付费/配 key 的用户 |

**注意**：Mode 2/3 都**不参与实时逐条判断**，只做后台的"规则挖掘"和"疑难样本复核"。实时判断永远走 Mode 1 的规则引擎。

## 二、本地大模型的三条技术路线（调研结论）

### 路线 A：Chrome 内置 AI（Gemini Nano + Prompt API / LanguageModel）

**是什么**：Google 把 Gemini Nano 内嵌进 Chrome，扩展可通过 `LanguageModel` API（前身 `window.ai`）调用，**本地推理、零云**。

**真实可用性（2026 年现状，务必注意条件）**：
- 2025 Google I/O 后进入 stable 通道，但**不等于默认可用**
- 前置门槛（全部满足才行）：
  1. Chrome 版本足够新（127+，推荐 138+ stable）
  2. 硬件达标（足够内存/磁盘/兼容 GPU）
  3. **用户已主动下载模型**（首次使用会触发下载，几百 MB）
  4. 不在不可用机型黑名单（部分 Mac 如 M4 早期有报告不可用）
- 模型**不可选**：只有 Gemini Nano，不能用别的

**检测与调用骨架**：
```ts
// 1. 检测可用性（必做，且要降级）
const avail = await LanguageModel.availability();
// 返回 'available' | 'downloadable' | 'downloading' | 'unavailable'
if (avail === 'unavailable') {
  // 用户环境不支持 → 降级到纯规则或提示启用
} else if (avail === 'downloadable') {
  // 可用但模型未下载 → 询问用户是否下载（几百 MB）
  const params = { initialPrompts: [...], monitor(m){...} };
  const session = await LanguageModel.create(params); // 触发下载
} else if (avail === 'available') {
  const session = await LanguageModel.create({ initialPrompts: [SYSTEM_PROMPT] });
  const stream = session.promptStreaming(userPrompt);
  // 收集流式结果
  session.destroy(); // 用完释放
}
```

**能力边界（Gemini Nano）**：
- 参数量级 ~1.8B-3B，轻量
- 文本分类、摘要、抽取都能做
- 中文支持有，但效果不如专门优化中文的小模型（如 Qwen）
- 速度：轻任务数百 ms 级

**适用判断**：✅ 可作为 Mode 2 的**可选实现之一**，但**不能作为唯一依赖**（可用性不稳），必须配降级。

### 路线 B：WebLLM（WebGPU + WASM，自选模型）✅ 推荐

**是什么**：开源 JS 引擎，用 WebGPU 在浏览器跑开源小模型（Qwen/Llama/Phi/Gemma），支持中英，模型可自选。

**为什么推荐做本地大模型的主力**：
- ✅ **模型可选**：能选中英优化的小模型（Qwen），对中文色情/博彩话术效果好
- ✅ 完全开源、零云、零 API key
- ⚠️ 要用户下载模型（几百 MB）、占内存（3-10GB）、需要 WebGPU

**模型选型建议**（垃圾评论分类任务）：

| 模型 | 体积 | 运行内存 | 中文 | 评价 |
|---|---|---|---|---|
| **Qwen3-0.6B / Qwen2.5-0.5B** | ~0.5-1GB | ~2-3GB | ✅ 强 | **首选**，体积小、中文好 |
| Qwen3-1.7B | ~1.5GB | ~4GB | ✅ 强 | 精度更高，适合愿意等的用户 |
| Gemma-3-270M | ~240MB | <1GB | 🟡 一般 | 最轻量，精度一般 |
| Phi-3.5-mini | ~2GB | ~5GB | 🟡 一般 | 英文强，中文弱 |

**在扩展里的架构（关键）**：
MV3 service worker 有生命周期限制且不能用 WebGPU，**必须用 offscreen document**：

```
content script (实时规则判断)
   ↓ sendMessage("classify-batch", samples)
service worker (中转 + 生命周期管理)
   ↓ chrome.offscreen API
offscreen document (持久 DOM + WebGPU)
   ↓ WebLLM Worker
   推理 → 返回 {sample, label, reason}
```

**最小代码骨架**（offscreen document 内）：
```ts
import { CreateMLCEngine } from '@mlc-ai/web-llm';

const engine = await CreateMLCEngine('Qwen2.5-0.5B-Instruct-q4f16_1-MLC', {
  initProgressCallback: (p) => { /* 下载进度 */ },
});

// 分类一批样本
async function classifySamples(samples: string[]) {
  const results = [];
  for (const text of samples) {
    const reply = await engine.chat.completions.create({
      messages: [
        { role: 'system', content: '判断下面推文评论是否为垃圾信息（色情/博彩/引流广告）。只输出 JSON：{"spam":true/false,"type":"...","reason":"..."}' },
        { role: 'user', content: text.slice(0, 200) },
      ],
      temperature: 0,
    });
    results.push(JSON.parse(reply.choices[0].message.content));
  }
  return results;
}
```

### 路线 C：Firefox 内置 AI（ONNX + Transformers.js）

- Firefox 基于 ONNX Runtime + Transformers.js，扩展可跑兼容模型
- 仅 Firefox 用户可用，覆盖面窄
- 作为兼容选项，不作为主力

### 三条路线选型结论

| 场景 | 推荐 |
|---|---|
| Mode 2 默认本地实现 | **WebLLM + Qwen2.5-0.5B**（offscreen document），中英好、可选模型 |
| 用户 Chrome 新 + 想零下载 | 检测到 Chrome 内置 AI 可用就优先用（省下载） |
| 兼容 Firefox | 用 Firefox 内置 Transformers.js |
| 都不可用 / 用户要最强 | 降级到 Mode 3 云端，或纯规则 |

## 三、核心流程：智能规则挖掘（你说的"检测新类型→整理规则"）

这是整个 AI 集成的**精华**。流程如下：

### 3.1 后台采样检测

```
评论区评论流
  ↓ 实时规则引擎判断
  ├─ 命中规则 → 折叠（Mode 1 实时处理）
  └─ 未命中，但【可疑度评分高】→ 进入"待复核样本池"
                                      ↓ 后台定时（每 N 分钟 / 累计 M 条）
                                      大模型批量分析（Mode 2 或 3）
                                      ↓
                                { 是新类型垃圾? → 提炼候选规则 }
                                      ↓
                                用户确认 → 写入规则库
                                      ↓
                                实时规则引擎立刻能过滤这种新垃圾 ✅
```

### 3.2 "可疑度评分"（决定哪些样本送大模型，省成本）

不是所有评论都送大模型（贵/慢），先用**廉价启发式**筛出可疑样本：

```ts
// lib/rules/suspicion.ts
function suspicionScore(text: string, ctx: Meta): number {
  let score = 0;
  // 规则没命中，但有垃圾结构特征
  if (/\+?\d[\s-]?v?x|威|号|@/.test(text)) score += 0.3;      // 联系方式引导
  if (/约|私|主?页| línea|telegram|line[:\s]/i.test(text)) score += 0.3; // 导流话术
  if (/(.)\1{3,}/.test(text)) score += 0.2;                  // 重复字符
  if (text.length < 30 && /[\u{1F300}-\u{1FAFF}]/u.test(text)) score += 0.2; // 短文+emoji
  if (ctx.accountAgeDays < 30) score += 0.2;                 // 新号
  if (ctx.followers < 10) score += 0.2;                      // 低粉
  return score;
}
// score >= 0.5 → 入待复核池
```

### 3.3 大模型提炼规则的 Prompt

让大模型从一批可疑样本里**归纳共性、产出可执行规则**，而不是逐条判断：

```ts
const RULE_MINING_PROMPT = `你是一个推特垃圾评论分析专家。下面是一批评分可疑、但未被现有规则命中的评论样本。
请分析它们是否属于同一种新的垃圾类型（如新型色情引流、博彩新话术等）。

如果是，提炼出能捕捉这种模式的【关键词/正则规则】，输出 JSON：
{
  "isNewPattern": true,
  "category": "porn|gambling|promo|spam|none",
  "suggestedRules": [
    { "type": "keyword", "value": "xxx", "match": "wordBoundary", "reason": "..." },
    { "type": "regex", "value": "\\bxxx\\b", "reason": "..." }
  ],
  "confidence": 0.0-1.0,
  "summary": "这批样本的共同特征是..."
}
如果只是正常评论（不是垃圾），isNewPattern=false。

样本：
${samples.map((s,i)=>`[${i}] ${s.slice(0,150)}`).join('\n')}`;
```

### 3.4 用户确认闭环（防误伤，关键）

大模型产出的规则**不能直接自动生效**（会误伤），必须走确认：

```
大模型提炼候选规则
  ↓
Popup/Options 弹出"发现 N 条疑似新型垃圾，建议添加 K 条规则"
  ↓ 展示：样本示例 + 建议规则 + 置信度
  ↓ 用户【采纳/编辑/拒绝】
  ↓
采纳 → 写入用户规则库（标记 source: 'ai-mined'）→ 实时层生效
拒绝 → 样本标记为"非垃圾"，避免再次送检（负反馈学习）
```

## 四、整体三模式架构图

```

│  实时层（content script，每条评论都过）                         │
│  ┌──────────────────────────────────────────────────────────┐│

│  │   内置规则 + 用户规则 + 云端词库 + AI挖掘规则               ││

│  │   → 未命中但可疑(suspicion≥0.5)：进待复核池                 ││

                              │ (待复核池累计/定时)
                              ▼

│  └─────────┬─────────┘  └────────┬─────────┘ └─────────────┘ │

└────────────────────────────┬─────────────────────────────────┘
                             ▼
┌──────────────────────────────────────────────────────────────┐
│  确认闭环（Popup/Options）                                      │
│  "发现新型垃圾，建议 N 条规则" → 用户采纳/编辑/拒绝 → 写入规则库 │
└──────────────────────────────────────────────────────────────┘
                             ▼
              实时层规则变强，下次自动过滤该类垃圾 ✅
```

## 五、用户设置（三模式开关）

```ts
interface AiSettings {
  // 实时层永远用规则，不可关
  // 智能层后端选择：
  backend: 'none' | 'local-chrome' | 'local-webllm' | 'cloud';

  // 本地
  localModel: 'qwen2.5-0.5b' | 'qwen3-0.6b' | 'gemma-270m' | 'chrome-nano';
  // 云端
  cloudProvider: 'deepseek' | 'openai' | 'custom';
  cloudApiKey: string;        // 仅本地存，不上传
  cloudEndpoint?: string;     // custom 用

  // 挖掘策略
  miningInterval: number;     // 多久跑一次（分钟），默认 30
  miningBatchSize: number;    // 每批样本数，默认 20
  suspicionThreshold: number; // 可疑分阈值，默认 0.5

  // 隐私
  uploadOnlyConfirmed: boolean; // 仅在用户主动触发时上传，默认 true
}
```

**默认值（开箱即用、零隐私顾虑）**：`backend: 'none'`（纯规则），用户想更智能再 opt-in 本地或云端。

## 六、可行性结论与风险

### 可行性
- ✅ **三模式架构完全可行**，且符合"本地优先、隐私优先"的最佳实践
- ✅ **"大模型挖规则、规则实时判断"是这类扩展用 AI 的最优雅姿势**，规避了大模型不适合实时的问题
- ✅ 浏览器本地大模型**真实存在**（Chrome 内置 AI / WebLLM / Firefox 内置），但都有可用性门槛，必须检测+降级

### 风险与对策

| 风险 | 对策 |
|---|---|
| Chrome 内置 AI 用户环境不支持 | `LanguageModel.availability()` 检测，不可用就降级到 WebLLM 或纯规则 |
| WebLLM 模型下载慢/占内存 | 明确告知用户体积，默认不下载，opt-in；提供最小模型选项（Gemma-270M） |
| WebGPU 不可用 | WebLLM 可降级 WASM（慢但能跑），或提示用户 |
| 云端 API 成本/配额 | 默认只挖规则不逐条判断，批量+定时，成本可控；用户自填 key |
| 大模型误挖规则导致误伤 | **必须用户确认闭环**，AI 规则标记 source 不自动生效 |
| 大模型把正常评论判为垃圾 | 负反馈学习（拒绝的样本不再送检），置信度门槛 |
| 隐私（上传文本） | 默认纯本地/纯规则；云端 opt-in；本地模式零网络；明确告知 |

### 落地优先级建议
1. **Phase 1 (MVP)**：只做 Mode 1 纯规则（先把骨架立住）
2. **Phase 2**：加 Mode 3 云端（实现简单，接个 API 即可）+ 规则挖掘闭环（产品差异化核心）
3. **Phase 3**：加 Mode 2 本地大模型（WebLLM offscreen，工程复杂度高放最后）+ Chrome 内置 AI 探测

> 理由：云端最简单见效（接 API），本地大模型工程复杂（offscreen+Worker+模型下载+WebGPU降级）放后面。规则挖掘闭环是产品差异化的灵魂，应该在 Phase 2 就有。

## 七、调研信息来源与不确定性说明

**已确认**：
- Chrome 内置 AI（Gemini Nano + Prompt/LanguageModel API）存在，2025 I/O 后进 stable，但有版本/硬件/模型下载门槛——来源：[Chrome Built-in AI 文档](https://developer.chrome.com/docs/ai/built-in)、[Prompt API 文档](https://developer.chrome.com/docs/ai/prompt-api)
- WebLLM 存在且成熟，支持 Qwen/Llama/Phi/Gemma，1-8B 天花板，~80% 原生速度——来源：[WebLLM GitHub](https://github.com/mlc-ai/web-llm)、[arXiv 论文](https://arxiv.org/html/2412.15803v2)
- Firefox 内置 AI 基于 ONNX+Transformers.js，扩展可用——来源：[Mozilla Blog](https://blog.mozilla.org/en/firefox/firefox-ai/running-inference-in-web-extensions/)
- 模型体积/内存数据来自社区实测（Reddit/技术博客），数量级准确

**不确定（需在开发时实测验证，文档中已标注）**：
- Chrome 内置 AI 在扩展 offscreen document 中调用的确切限制（搜索显示 service worker 受限，需 offscreen——这是 MV3 跑持久 DOM/WebGPU 的标准答案，但 LanguageModel API 是否同样受限需实测）
- WebLLM 模型能否预打包进扩展包随分发（受扩展体积限制 + CDN 下载更现实，默认走 CDN）
- Gemini Nano 对中文色情/博彩话术的实际分类精度（需实测，建议优先 Qwen）
- WebLLM 在 offscreen + Web Worker 里的确切推理速度（需实测）

> 这些都是实现细节，不影响"三模式架构"的整体可行性判断。开发到 Phase 3 时做技术 spike 验证即可。
